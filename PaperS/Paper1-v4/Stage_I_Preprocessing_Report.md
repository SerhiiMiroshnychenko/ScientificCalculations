# Звіт про методологію попередньої обробки даних (Stage I): Формування набору "cleanest_data.csv"

## 1. Архітектурний контекст
Процес підготовки даних для дослідження прогнозування успішності замовлень у B2B-сегменті розділений на два ключові етапи. **Stage I** (Engineering & Filtering) є фундаментом, на якому базується подальше моделювання. Результатом цього етапу є файл `cleanest_data.csv`, який містить 25 ретельно відібраних ознак. Важливо підкреслити, що видалення нерелевантних даних та запобігання "витоку цільової змінної" (target leakage) відбувається саме на цьому етапі, ще до початку статистичного аналізу в Jupyter-ноутбуках.

## 2. Основний інструментарій
"Серцем" першого етапу є Python-скрипт:
`D:\PROJECTs\MY\ScientificCalculations\SC\ScientificCalculations\Predict\Порівняння\build_b2b_from_pgadmin_export.py`

Цей скрипт автоматизує трансформацію сирого SQL-експорту з pgAdmin у структурований аналітичний набір даних, реалізуючи логіку, описану в проектному гайді `pgadmin_to_b2b_minimal_b2b_guide.md`.

## 3. Запобігання витоку цільової змінної (Target Leakage)
Одним із найскладніших аспектів Stage I є побудова історичних агрегатів по партнерах (`partner_*`). Для забезпечення чистоти експерименту скрипт реалізує **Temporal Leakage Prevention**.

### Механізм захисту:
При розрахунку таких характеристик, як `partner_success_rate` або `partner_avg_amount` для конкретного замовлення, дані *поточного* замовлення не враховуються. Це досягається шляхом віднімання значень поточного замовлення з кумулятивних сум за допомогою логіки:
```python
df["partner_total_messages"] = (cum_messages_all - df["order_messages"])
df["_amount_prev_all"] = (cum_amount_all - df["order_amount"])
```
Таким чином, модель на кожному кроці "бачить" тільки те, що було відомо системі *до* моменту створення поточного замовлення.

## 4. Точний список видалених ознак
У процесі переходу від розширеного датасету (`data_collector_extended.csv`) до очищеного (`cleanest_data.csv`) видаляються наступні категорії даних:

### А. Ідентифікатори та технічні ключі (Leakage Risk)
- `id`, `order_id`, `partner_id` — видаляються як ознаки, оскільки вони можуть бути використані моделлю для "запам'ятовування" конкретних випадків замість вивчення закономірностей.
- `customer_id` — дублює інформацію.

### Б. Нерелевантні бізнес-ознаки
Ці ознаки були виключені через низьку предиктивну здатність у контексті B2B-замовлень або через надмірну зашумленість:
- `sales_team` (відділ продажів)
- `customer_category` (категорія клієнта)
- `customer_country` (країна клієнта)
- `product_categories` (категорії товарів)
- `payment_term` (умови оплати)
- `delivery_method` (метод доставки)
- `processing_time_hours` — видалено, оскільки це значення стає відомим лише *після* завершення замовлення, що є прямим leakage.

### В. Технічні змінні (Intermediate variables)
Видаляються всі проміжні стовпці, що починаються з підкреслення (наприклад, `_amount_prev_all`, `_success_flag`), які використовувалися лише для розрахунку історичних агрегатів.

## 5. Поріг варіативності (Near-Zero Variance)
На етапі Stage I застосовується логіка фільтрації ознак з низькою варіативністю. Ознаки, у яких понад **95%** значень є ідентичними після очищення, вважаються неінформативними. 

Наприклад, якщо після виключення даних за 2025 рік певна категорія `source` представлена лише в поодиноких випадках, вона не потрапляє до фінального переліку `cleanest_data.csv`. Це дозволяє уникнути проблеми "розрідженості" даних (sparsity) при подальшому кодуванні.

## 6. Висновок
Файл `cleanest_data.csv` потрапляє на етап `DATA-PREPROCESSING` уже в максимально валідованому стані. Всі операції, пов'язані з бізнес-логікою Odoo, захистом від витоку даних та видаленням технічного шуму, завершуються в межах скрипта `build_b2b_from_pgadmin_export.py`. Це забезпечує високу якість вхідних даних для статистичного аналізу та машинного навчання.
